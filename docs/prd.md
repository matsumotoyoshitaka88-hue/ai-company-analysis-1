## 1. 課題

### 背景
企業の管理職は、自社の経営状況（財務健全性、競合ポジション、DX成熟度など）を体系的に把握する手段が限られている。

### 現状の課題
- **コンサルは高額**: PEファンドやコンサルファームのDDは数百万〜数千万円かかり、中堅企業の管理職が気軽に利用できない
- **自力での分析は困難**: 有報やIR資料は公開されているが、読み解きに専門知識が必要で、管理職の多くは十分に活用できていない
- **既存ツールは断片的**: 財務分析ツール、業界レポート等は存在するが、DD的な総合診断を提供するものがない
- **意思決定の質に影響**: 自社の立ち位置が不明なまま経営判断を行うリスクがある

### ターゲットユーザー
- **主要ターゲット**: 中堅企業（売上10億〜500億円）の管理職・経営企画担当
- **ペルソナ**: 40-50代、経営会議で自社の状況を報告する立場、データ分析の専門知識は限定的

---

## 2. 目的・ゴール

### ビジネス目的
無料のAI企業診断を提供し、診断利用者を本業サービスのリード（見込み顧客）として獲得する。

### プロダクト目標
| 目標 | 指標 | ターゲット |
|------|------|-----------|
| 診断の価値提供 | 診断完了率 | 70%以上 |
| ユーザー満足 | 診断後NPS | 30以上 |
| リード獲得 | 診断→本業CTA クリック率 | 5%以上 |
| 商談創出 | CTA→問い合わせ転換率 | 1%以上 |

### 非ゴール（スコープ外）
- 有料サービスとしてのマネタイズ
- コンサルファームを完全に代替する深度のDD
- 非公開情報（社内機密データ）を扱う機能（v1では対象外）

---

## 3. ソリューション概要

PEファンドのDDをAIで簡易化した無料の企業診断サービス。企業名を入力するだけで、公開データを自動収集し、AIが総合診断レポートを生成する。

### コアコンセプト
```
企業名入力 → 公開データ自動収集 → AI分析 → 診断レポート → 本業サービスへの導線
```

---

## 4. 機能要件

### P0: Must Have（MVP）

#### 4.1 公開データ自動収集エンジン
- 企業名（または証券コード）を入力すると、以下を自動収集:
  - 有価証券報告書（EDINET）
  - 企業ニュース（直近1年）
  - 業界データ
- **対象**: まず上場企業のみ

#### 4.2 AI診断レポート生成
LLMを活用し、以下の診断セクションを自動生成:

| セクション | 内容 |
|-----------|------|
| エグゼクティブサマリー | 経営状況の総合評価（1ページ） |
| 財務診断 | 収益性・安全性・効率性のスコアリング、業界平均比較 |
| 競合ポジション | 同業他社3-5社との比較（売上成長率、利益率等） |
| DX成熟度 | IT投資・デジタル関連の取り組み評価 |
| リスク・機会分析 | 主要リスクと成長機会の洗い出し |
| 総合スコア | 100点満点の総合評価（信号機方式で可視化） |

#### 4.3 経営ダッシュボード型UI
- 信号機方式（赤黄緑）で各領域の状態を直感表示
- 段階的開示: サマリー → 各セクション詳細への掘り下げ
- モバイル対応のレスポンシブデザイン

#### 4.4 リード獲得導線
- 診断レポート末尾に本業サービスへのCTA
- 診断結果に基づく「次のアクション提案」に本業サービスを自然に組み込む
- メールアドレス登録で詳細レポートをPDF送付（リード情報取得）

### P1: Should Have（v1.1）

#### 4.5 AI対話型診断
- チャットボット形式で追加情報をヒアリング
- 非公開情報（社内データ）を対話で収集し、診断精度を向上
- 「話すだけ」で診断が進む体験

#### 4.6 業界特化診断モデル
- 製造業、小売業、SaaS/IT企業向けの特化ロジック
- 業界固有のKPI・ベンチマークを使用

### P2: Nice to Have（将来）

#### 4.7 競合ベンチマーク詳細版
- 非上場企業を含む競合分析
- 市場シェア推定

#### 4.8 データ連携
- 会計ソフト（freee, マネーフォワード）API連携
- SaaS（Salesforce等）連携によるリアルタイム診断

## 5. 技術要件

### アーキテクチャ方針

| レイヤー | 技術方針 |
|---------|---------|
| フロントエンド | Webアプリケーション（レスポンシブ） |
| バックエンド | API サーバー + 非同期処理（レポート生成） |
| AI/LLM | Claude API（診断レポート生成） |
| データ収集 | EDINET API、ニュースAPI、Webスクレイピング |
| データベース | 企業データキャッシュ、ユーザー情報、診断履歴 |

### パフォーマンス要件
- 診断レポート生成: 3分以内（目標1分以内）
- 同時診断リクエスト: 100件/時
- データ鮮度: 公開データは日次更新

### セキュリティ要件
- ユーザー入力情報の暗号化
- LLMへの送信データは公開情報のみ（v1）
- 個人情報保護法・プライバシーポリシーの整備

## 8. リリース計画

### Phase 1: MVP（ローンチ後0-3ヶ月）
- 上場企業のみ対象
- 公開データのみで診断
- 基本的なダッシュボード + PDF出力
- リード獲得導線

### Phase 2: 拡張（3-6ヶ月）
- AI対話型診断の追加
- 業界特化モデル（製造業・SaaSから開始）
- 診断レポートの品質改善（フィードバック反映）

### Phase 3: スケール（6-12ヶ月）
- 非上場企業対応
- データ連携（会計ソフト・SaaS）
- 競合ベンチマーク詳細版

## 9. 前提条件・依存関係

### 前提条件
- EDINET APIで有報データを取得可能
- LLM（Claude API）の出力品質がDD基礎レベルに達する
- 管理職がAI診断を信頼し利用する

### 依存関係
- Claude API の利用枠・コスト
- EDINET API の利用制限
- 本業サービス側のリード受け入れ体制

### リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| LLMの分析精度が不十分 | 高 | 専門家レビューによるプロンプト改善、人間レビューの併用 |
| 公開データだけでは診断が浅い | 高 | P1のAI対話型診断で非公開情報を補完 |
| 管理職がAI診断を信頼しない | 中 | 「AIアシスト×専門家監修」のブランディング |
| 競合サービスの出現 | 中 | 業界特化・本業との連携で差別化 |
